# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

ZENITH is a cosmetics POS (Point of Sale) system for a Mexican cosmetics business. This is a **context-driven project** — code is generated by AI guided by a set of context files that serve as the single source of truth. There is no traditional build system, test suite, or package.json yet.

**Language**: Spanish (Mexico) for all UI text. No unnecessary anglicisms, but domain terms like "stock", "display", "gloss" are kept as-is.

**Currency**: MXN (Mexican Peso). Format: `$1,805.00 MXN`

## Context System Architecture

All context files live in `/Context/`. Each file governs a specific domain and must be consulted before generating related code. **These files are source of truth, not suggestions.**

| File | Format | Consult When |
|------|--------|--------------|
| `zenith-context-system.md` | MD | Understanding the overall system and what context files exist |
| `zenith-design-system.xml` | XML | Generating any UI — colors, typography, spacing, shadows, motion (180+ tokens) |
| `ui-components.xml` | XML | Building any UI component (buttons, inputs, cards, modals, sidebar, search-bar, cart, swatches, etc.) |
| `iconography.xml` | XML | Using any icon — Lucide Icons library, 90+ mapped icons, stroke/size/color rules |
| `pos-data-model.xml` | XML | Any screen that displays/manipulates data, DB schema, CFDI 4.0 fiscal rules |
| `ui-copy.md` | MD | Writing any user-facing text, microcopy patterns, glossary, data formatting |
| `pos-screens.md` | MD | Designing or generating any screen — 10 screens with ASCII layouts and responsive rules |
| `pos-workflows.md` | MD | Programming business logic — 10 flows with error handling and role permissions |

### Which files to load per task

- **Generating a screen**: `design-tokens` + `ui-components` + `pos-screens` + relevant workflow from `pos-workflows`
- **Writing UI copy**: `ui-copy.md`
- **Data model / DB work**: `pos-data-model.xml`
- **Using icons**: `iconography.xml`

## Design Token Rules

All UI code must use **only** the predefined design tokens from `zenith-design-system.html`. Never use raw hex values, pixel sizes, or hardcoded timing — always reference tokens.

- **Token pattern**: `--{category}-{step}` (e.g., `--rose-500`, `--space-4`, `--shadow-sm`)
- **Color palettes**: Rose (primary), Teal (secondary), Neutral (foundation), Blush (accent) + 4 semantic (success, warning, error, info)
- **Fonts**: Zodiak (serif, display/headings) and Plus Jakarta Sans (sans-serif, body/UI) — both from Fontshare
- **Spacing**: 4px base unit, scale from `--space-1` (4px) to `--space-32` (128px)
- **Breakpoints**: sm (640px), md (768px), lg (1024px), xl (1280px), 2xl (1536px)

## Domain: Cosmetics POS

Key domain concepts that affect code:

- **ProductVariant** is the sellable unit (not Product). Each variant has shade name, shade hex (for UI swatches), finish (matte/satin/shimmer/gloss/metallic/cream), size, price, barcode.
- **Sale** supports split payments (cash + card).
- **CashRegister** is session-based with opening/closing/reconciliation.
- **Customers** are optional per sale.

## UI Copy Conventions

- **Primary buttons**: Verb + direct object → "Cobrar $1,805" (not "Procesar pago")
- **Confirmations**: Result + brief detail → "Venta registrada #0042" (not "Operación exitosa")
- **Validation errors**: What's missing + how to fix → "Agrega al menos 1 producto" (not "Error: carrito vacío")
- **Empty states**: What will appear here + next action
- **Date format**: `18 mar 2026` — **Time**: `2:30 pm` — **Phone**: `+52 33 1234 5678`

## Fundamental Rule

If a component, color, flow, or term is not defined in the context files, **define it in the corresponding context file first, then implement**. Never the other way around.
